# 要件定義書 — pitwu.com リニューアル

**バージョン**: 0.4
**作成日**: 2026-04-14
**ステータス**: 承認済み（2026-04-14）— v0.4 追記（カレンダー・スケジュール管理）

---

## 概要

pitwu.com を WordPress（さくらサーバー）から **Astro + microCMS + Cloudflare Pages** へ移行する。Git・LLMとの親和性を高め、運用コストを削減しながら、既存コンテンツとSEO評価を引き継ぐ。

---

## 目的・背景

| 課題 | 解決方針 |
|------|----------|
| WordPressはGit管理・LLMとの相性が悪い | コードはGit管理、コンテンツはheadless CMSへ分離 |
| さくらサーバーの固定費・保守負担（DBパッチ等） | Cloudflare Pages（無料枠）へ移行し固定費をほぼゼロ化 |
| 表示速度・Core Web Vitalsの改善余地 | Astroの静的生成でパフォーマンス最大化 |

---

## ユーザー

- **一般閲覧者**: 記事を読む、全文検索する、既存コメントを閲覧する
- **認証ユーザー**: Cloudflare Zero Trust で許可された特定メールアドレス/ドメインの利用者（特定コンテンツへのアクセス権を持つ）
- **管理者（エンジニア）**: microCMS で記事を管理し、Gitでコードをデプロイする

---

## 機能要件

### コンテンツ表示
- [ ] **REQ-01**: 約2,000件の記事をAstroで静的レンダリングして表示する
- [ ] **REQ-02**: 約20枚の固定ページをAstroプロジェクト内（`.astro` / Markdown）で管理・表示する
- [ ] **REQ-03**: 記事ページに既存のWordPressコメントを静的表示する（新規コメント投稿機能は不要）

### 全文検索
- [ ] **REQ-04**: Pagefind をビルドプロセスに組み込み、サイト内全文検索機能を提供する
- [ ] **REQ-05**: 検索はクライアントサイドで完結し、外部APIへの依存・追加コストを持たない

### 認証・アクセス制御
- [ ] **REQ-06**: Cloudflare Zero Trust (Access) を用いて、特定のメールアドレスまたはドメインを持つユーザーのみが特定パス（または全体）へアクセスできるよう制限する
- [ ] **REQ-07**: アプリケーションコード内に認証ロジックを実装しない（インフラ層で完結）

### CMS・コンテンツ管理
- [ ] **REQ-08**: 記事コンテンツは microCMS で管理し、Astroはビルド時にAPIからデータを取得する
- [ ] **REQ-09**: 固定ページ（更新頻度低）はGitリポジトリ内のファイルで管理する

### CI/CD・デプロイ
- [ ] **REQ-10**: GitHubへのプッシュをトリガーにCloudflare Pagesへ自動デプロイされる

### カレンダー・スケジュール管理
- [ ] **REQ-11**: よさこいチーム「夢源風人」の活動予定を Google Calendar で管理する（イベント用・練習用の2カレンダーを使い分け）
- [ ] **REQ-12**: サイト上の `/schedule/` ページで Google Calendar の予定を閲覧できるようにする
- [ ] **REQ-13**: Google Calendar の CRUD 操作を CLI スクリプト（`npx tsx scripts/gcal.ts`）で行えるようにする（追加時の重複チェック、終日/時刻指定の自動判定を含む）
- [ ] **REQ-14**: Claude Code の `gcal-agent` サブエージェントを通じて、自然な会話からカレンダー操作ができるようにする（運用補助）
- [ ] **REQ-15**: Google Calendar API 認証はサービスアカウント（`.env` の `GOOGLE_SERVICE_ACCOUNT_EMAIL` / `GOOGLE_PRIVATE_KEY`）を用い、リポジトリに秘密情報を含めない

## 移行要件

- [ ] **MIG-01**: WordPressのXMLエクスポートから記事データを取得する
- [ ] **MIG-02**: スクリプト（`wordpress-export-to-markdown` 等）でMarkdown/JSONへ一括変換する
- [ ] **MIG-03**: microCMS マネジメントAPIを使用するスクリプトで記事・画像を一括登録する
- [ ] **MIG-04**: 記事内画像のURLをWordPress時代のパスからmicroCMS（`images.microcms-assets.io` 等）へ一括置換する
- [ ] **MIG-05**: `_redirects` ファイルでパーマリンク構造が変わる場合のリダイレクトを設定し、SEO評価を引き継ぐ

## 非機能要件

| 項目 | 要件 |
|------|------|
| パフォーマンス | Core Web Vitals でGood判定（Pagespeed Insights）を目標 |
| SEO | 既存URLのパーマリンク維持、またはリダイレクトによる評価引き継ぎ。`<title>` / `description` のパターンは `docs/design.md` を参照。ビルド時に XML サイトマップ（`@astrojs/sitemap`）を生成し、`public/robots.txt` で sitemap の URL を宣言する（詳細は `docs/design.md`） |
| コスト | ホスティング費用を現在のさくらサーバー費用以下（できれば無料枠内）に抑える |
| 保守性 | DBやサーバーの保守作業をなくす（フルマネージド環境） |
| Git親和性 | コードはすべてGitで管理し、LLMによる補助開発が可能な構成とする |

### 表記・メタ情報

- **用語**: 年度ごとの楽曲・振付・紹介のまとまり（`/discography/`）はサイト上 **「作品」** と表記する。企業・施設への訪問出演など **依頼・出演の文脈** では **「演舞」** を用いる（例: 演舞依頼ページ、訪問演舞）。
- **SEO**: `<title>` は短くブランド・ページ種別を示し、補足・情緒は `meta description` に載せる。具体パターン・例は `docs/design.md` の「SEO・表記・メタ情報」を参照する。
- **XML サイトマップ**: 本番の絶対 URL（`astro.config.mjs` の `site`）を前提に、`npm run build` で `sitemap-index.xml` 等を `dist/` に出力する。`robots.txt` の `Sitemap:` は本番ドメインと一致させる。

## 検索機能の改善要件（v1.0 — 2026-04-17）

**バージョン**: v1.0
**作成日**: 2026-04-17
**ステータス**: 承認済み（2026-04-17）

---

### 概要

Pagefind の CJK bigram OR マッチにより、ひらがな1〜2文字の入力で数百件ヒットする過検索（ノイズ）と、読み仮名でのヒット漏れ（例:「たいが」で「大雅」がヒットしない）が同時に発生している。ビルド時の形態素解析と手動揺れ辞書を組み合わせてインデックスを補強し、検索精度を改善する。

---

### 現状の把握

| 項目 | 現状 |
|------|------|
| 検索エンジン | Pagefind v1.3.0（`package.json` 参照） |
| インデックス設定 | `data-pagefind-body` を `.article-body` div に付与のみ（設定ゼロ） |
| CJK マッチ挙動 | Pagefind デフォルトの bigram OR マッチ（2文字以上の部分一致） |
| 既知の問題 | ひらがな単体入力で大量ヒット（例:「か」で500件超）／読み仮名で漢字表記がヒットしない |
| 検索対象 | 活動記事のみ（`/[year]/[month]/[slug]/`）。discography・固定ページは対象外 |
| 検索 UI | `src/components/Search.astro` — Pagefind UI デフォルト（CSS トークンのみカスタマイズ） |

---

### ユーザー要件

- **B1（読み仮名ヒット）**: 人名・地名等の固有名詞を読み仮名で検索したとき、その漢字表記を含む記事がヒットする
- **B2（手動揺れ辞書）**: 「たいが → 大雅」のようなサイト固有の表記揺れを JSON ファイルに手動登録でき、ビルド時にインデックスへ反映される
- 上記 B1・B2 は一括実装（Phase 分割なし）
- 結果カードは Pagefind UI デフォルト（タイトル＋本文抜粋）のままとする
- 並び順は関連度順のみ（新着順トグル等は不要）
- 辞書への個人プライベート情報は含めない運用とする（暗号化不要）

---

### 技術方針

| # | 方針 |
|---|------|
| B1 | ビルド時に kuromoji.js で記事タイトル・本文の固有名詞を形態素解析し、読み仮名（カタカナ）を抽出する。抽出した読み仮名を、対象記事ページの `data-pagefind-body` 付き hidden div に追記し、Pagefind がインデックス化できるようにする |
| B2 | `src/data/search-aliases.json`（または `docs/search-aliases.json`）に `{ "よみ": ["漢字表記1", "漢字表記2"] }` 形式の手動揺れ辞書を置く。ビルド時スクリプトが同ファイルを読み込み、エイリアスを対応ページの hidden div に埋め込む。AI および人間による編集を前提とし、JSON は平易な構造を維持する |
| 共通 | hidden div への追記はビルド時の後処理スクリプト（`scripts/inject-search-hints.ts` 相当）として実装する。kuromoji 辞書はビルド時のみ使用し、ブラウザへは配信しない |

---

### 受け入れ条件

- [ ] **AC-01**: 「たいが」で検索したとき、「大雅」を含む記事が上位10件以内にヒットする
- [ ] **AC-02**: 「やまもと」で検索したとき、「山本」を含む記事がヒットする
- [ ] **AC-03**: 「か」など無関係な単一文字で検索したとき、ヒット件数が50件以下である（関連記事の取りこぼしなし）
- [ ] **AC-04**: スペース区切り（例:「た い が」）での検索でも正常にヒットする
- [ ] **AC-05**: discography ページ・固定ページが検索結果に表示されない（現状維持）
- [ ] **AC-06**: 検索応答時間が 100ms 以下である
- [ ] **AC-07**: ブラウザへの追加ロードが +150KB（gzip 後）以下である
- [ ] **AC-08**: ビルド時間の増加が +60秒以内である
- [ ] **AC-09**: Cloudflare Pages 無料枠内で完結し、外部サービス依存がない
- [ ] **AC-10**: `src/data/search-aliases.json` が存在し、AI・人間が編集可能な平易な JSON 形式である
